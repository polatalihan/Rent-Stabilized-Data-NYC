# NYC Rent-Stabilized Housing Tax Bill Scraper

A Python notebook for extracting reported rent-stabilized apartment counts from New York City Department of Finance property-tax statements by Borough-Block-Lot (BBL).

The notebook downloads a tax-statement PDF for each BBL, searches for the **Rent Stabilization–Chg** line, extracts the associated apartment count, and saves the results to a CSV. It uses parallel requests, periodic checkpointing, and automatic resume functionality for larger BBL lists.

## Repository contents

- `nyc_rent_stabilized_tax_bill_scraper.ipynb` — scraping and PDF-parsing workflow
- `bbls.csv` — user-provided input list with a required `bbl` column
- Output CSV files generated by the notebook

## Required input

Create a CSV named `bbls.csv`, or update the notebook to point to another filename. The file must contain a column named `bbl` with one 10-digit BBL per row.

```csv
bbl
1002470001
1001007502
1009720001
```

## User settings

In **Section 2: User settings**, edit these three variables:

```python
INPUT_CSV = "bbls.csv"
STATEMENT_DATE = "20211120"
OUTPUT_CSV = "rent_stabilized_counts_2021_q3.csv"
```

- `INPUT_CSV` identifies the BBL list.
- `STATEMENT_DATE` identifies the requested tax statement and must use `YYYYMMDD` format.
- `OUTPUT_CSV` identifies the results file. A date or quarter should be included in the filename so separate runs are not overwritten or mixed together.

## Tax-bill frequency and statement selection

NYC property-tax bills are typically issued four times per year on a quarterly schedule. Some properties, however, participate in a monthly payment program and may have monthly-program bills listed on the tax-bills page. Identifying those properties requires reviewing the available statements and analyzing the billing pattern rather than assuming every BBL follows the same quarterly schedule.

For the underlying housing analysis, the first- and third-quarter statements were scraped separately for each BBL. The two results were then merged, and the highest reported rent-stabilized apartment count was retained. This approach helps address cases in which one selected statement does not display a rent-stabilization charge or reports no rent-stabilized units while another statement does.

Because the rent-stabilized apartment count is derived from property-owner reporting reflected in the tax statement, checking multiple billing periods provides an additional quality-control step. A value appearing in a tax bill should not automatically be treated as independently verified regulatory data.

## Recommended multi-quarter workflow

1. Run the notebook for the first selected statement date and save the results to a quarter-specific CSV.
2. Change `STATEMENT_DATE` and `OUTPUT_CSV` for the second statement date.
3. Run the notebook again.
4. Join the two files by `bbl`.
5. Calculate the maximum non-null rent-stabilized apartment count across the two statement fields.
6. Review unusual differences, missing PDFs, zero values, and properties that appear to use monthly-program bills.

Example filenames:

```text
rent_stabilized_counts_2021_q1.csv
rent_stabilized_counts_2021_q3.csv
rent_stabilized_counts_2021_merged.csv
```

## Output fields

The notebook produces:

| Field | Description |
|---|---|
| `bbl` | Ten-digit Borough-Block-Lot identifier |
| `rent_stabilized_apartments` | Extracted apartment count associated with the rent-stabilization charge |
| `status` | Processing result, such as `ok`, `no_pdf`, or `no_rs_lines` |

A missing statement and a statement with no detected rent-stabilization line are recorded separately. These records should be reviewed before analysis.

## Running the notebook

Open the notebook in Jupyter Notebook, JupyterLab, or another compatible environment and run the cells in order. The dependency cell installs the required packages:

- `pandas`
- `requests`
- `pdfplumber`
- `pypdf`
- `tqdm`

The notebook uses multiple workers to accelerate requests. Reduce the `WORKERS` setting if the statement service becomes unstable or returns incomplete files.

## Resume behavior

The scraper saves checkpoints to the selected output file. When the notebook is rerun with the same `OUTPUT_CSV`, BBLs already present in that file are skipped.

Use a new output filename whenever the statement date changes. Otherwise, results from the earlier statement may be mistaken for completed records from the new run.

## Data limitations

- A requested statement may not exist for every BBL and date.
- PDF layouts and text extraction can vary.
- A missing rent-stabilization line does not necessarily prove that a property has no rent-stabilized apartments.
- Monthly-program properties may require additional statement review.
- Reported counts can differ between statements.
- The values reflect information reported through the property-tax billing process and should be quality-checked against additional periods or sources when used for research or policy analysis.

## Responsible use

Use reasonable request concurrency and avoid repeatedly downloading the same statements unnecessarily. Large runs should be checkpointed, reviewed for errors, and documented with the exact statement dates used.
